# CMDLY Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Build CMDLY, an AI-piloted infrastructure administration dashboard where a natural-language chat executes concrete actions (Terraform, Ansible, SSH, Proxmox/Wazuh/Zabbix APIs) via LLM function calling, guarded by RBAC, Zod validation, confirmations, rate limiting and an append-only audit log.

**Architecture:** Next.js 16 App Router (RSC + client components). API routes drive better-auth sessions, a multi-provider LLM abstraction (OpenAI/Anthropic/Ollama), and a tool executor that wraps every one of 14 tools with a guardrail pipeline. Secrets are AES-256-GCM encrypted at rest with a key derived from `BETTER_AUTH_SECRET`. PostgreSQL via Drizzle ORM. Chat streams over SSE using `EventSource` (POST to create a run, GET to stream it).

**Tech Stack:** Next.js 16.2, TypeScript strict, Tailwind CSS 4.3 (`@theme` in CSS), shadcn/ui, Drizzle ORM + PostgreSQL, better-auth 1.6, Framer Motion, Lucide, Zod, Recharts, openai / @anthropic-ai/sdk / ollama SDKs, node-ssh.

## Global Constraints

- Next.js 15+ App Router with Server Components by default; client components only where interactivity is required. Actual pinned version: Next.js 16.2.
- TypeScript strict mode — zero implicit `any`, zero `@ts-ignore`.
- Tailwind CSS 4.3.x — configuration via `@theme` in `src/app/globals.css`, NOT `tailwind.config.ts`. There is no Tailwind v5.
- Drizzle ORM only (never Prisma). Better-auth only (never NextAuth).
- Zod validates ALL external input: API route bodies, tool params, forms.
- Every shell/CLI invocation uses `execFile` with an argument array — never string interpolation into a shell.
- Every secret stored in the DB is AES-256-GCM encrypted via `src/lib/crypto.ts`. Never store secrets in plaintext.
- Every tool invocation passes RBAC (`canExecuteTool`), rate limiting, and writes an audit row both before and after execution.
- Audit log is append-only: INSERT only, never UPDATE or DELETE.
- Error handling uses typed discriminated unions, not thrown exceptions, across tool results and provider events.
- No `console.log` in production code — use the structured logger `src/lib/logger.ts`.
- Every user action ends in a success/error toast. Component entrances use subtle Framer Motion (fade + slide, 200ms). Responsive (sidebar collapse, full-screen chat). Accessibility: visible focus, aria-labels, AA contrast.
- Commit per task using Conventional Commits.

### Palette (design tokens, declared in `globals.css` `@theme`)

- `--color-background: #0A0A0B`, `--color-card: #111113`, `--color-border: #27272A`
- `--color-foreground: #EDEDED`, `--color-muted: #A1A1AA`
- `--color-navy: #1A365D` (primary), `--color-orange: #DD6B20` (accent/CTA/focus rings)
- `--color-success: #22C55E`, `--color-warning: #F59E0B`, `--color-danger: #EF4444`
- Radii rounded-lg/xl, soft shadows, 200-300ms transitions, occasional navy→orange gradients, no neon.

### Roles & guardrail constants

- `ROLE_HIERARCHY = { viewer: 0, operator: 1, admin: 2 }`.
- Rate limits per user, 1-minute sliding window: read = unlimited, modify = 5/min, destroy = 1/min.
- Confirmation-required tools: `restart_service`, `rollback`, `destroy_vm`, `stop_service`. `destroy_vm` additionally requires re-typing the VM name.
- Session lifetime 24h with refresh. Login lockout after 5 failed attempts for 15 min.

## File Structure

The full target tree is defined in the spec (`docs/superpowers/specs/2026-07-07-cmdly-spec-initiale.md`). Key responsibilities:

- `src/lib/crypto.ts` — AES-256-GCM `encrypt`/`decrypt`, key derived from `BETTER_AUTH_SECRET`. One responsibility: reversible secret storage.
- `src/lib/logger.ts` — structured JSON logger (`info`/`warn`/`error`/`debug`). Replaces all `console.*`.
- `src/lib/db/schema.ts` — all Drizzle tables. `src/lib/db/index.ts` — the drizzle client singleton.
- `src/lib/auth/config.ts` — better-auth server instance. `permissions.ts` — RBAC pure functions. `ldap.ts` — optional LDAP bind.
- `src/lib/config.ts` — load/save `infrastructureConfig` singleton, decrypting secrets on read, encrypting on write.
- `src/lib/llm/{index,openai,anthropic,ollama,streaming}.ts` — provider abstraction + implementations + SSE helpers.
- `src/lib/tools/registry.ts` — the 14-tool catalogue. `executor.ts` — dispatcher + guardrail pipeline. One file per tool.
- `src/lib/{proxmox,wazuh,zabbix,ssh,terraform,ansible}.ts` — infra clients. Each isolates one external system.
- `src/lib/{rateLimit,audit}.ts` — guardrail helpers backed by DB tables.
- `src/app/api/**` — route handlers. `src/middleware.ts` — route protection.
- `src/components/{chat,dashboard,onboarding,ui}/**` — UI, split per the spec tree.
- `src/hooks/{useChat,useConfirm,useToast}.ts` — client orchestration.
- `src/types/{tools,llm,auth}.ts` — shared type contracts.

Files that change together live together; each file has one responsibility. Prefer focused files over large ones.

import { createCipheriv, createDecipheriv, randomBytes, scryptSync } from "node:crypto";

function key(): Buffer {
  const secret = process.env.BETTER_AUTH_SECRET;
  if (!secret) throw new Error("BETTER_AUTH_SECRET is required for encryption");
  return scryptSync(secret, "cmdly-secret-salt", 32);
}
export function encrypt(plain: string): string {
  const iv = randomBytes(12);
  const cipher = createCipheriv("aes-256-gcm", key(), iv);
  const enc = Buffer.concat([cipher.update(plain, "utf8"), cipher.final()]);
  return Buffer.concat([iv, cipher.getAuthTag(), enc]).toString("base64");
}
export function decrypt(payload: string): string {
  const raw = Buffer.from(payload, "base64");
  const iv = raw.subarray(0, 12);
  const tag = raw.subarray(12, 28);
  const enc = raw.subarray(28);
  const decipher = createDecipheriv("aes-256-gcm", key(), iv);
  decipher.setAuthTag(tag);
  return Buffer.concat([decipher.update(enc), decipher.final()]).toString("utf8");
}
```

export type Role = "viewer" | "operator" | "admin";
export interface ExecuteOutcome {
  status: "success" | "error" | "denied" | "confirm_required";
  result?: ToolResult;
  reason?: string;                 // for denied
  confirm?: { action: string; params: Record<string, unknown>; requireTyping?: string }; // for confirm_required
}
```

- `audit.ts` produces `audit(row): Promise<void>` — INSERT only into `auditLog`; action ∈ `tool_call_attempted|tool_call_succeeded|tool_call_failed|tool_call_denied`.
- `registry.ts` produces `TOOLS: Record<string, Tool>` and `toolCatalogueForLLM(role): LLMTool[]` (only tools the role may call, mapped from Zod to JSON Schema via `zod-to-json-schema`... use a small hand-rolled converter or add the dep).
- `executor.ts` produces `executeTool(name, rawParams, ctx, opts?: { confirmed?: boolean }): Promise<ExecuteOutcome>` implementing the pipeline: (1) audit `attempted`; (2) tool exists else error; (3) `canExecuteTool` else audit `denied` + return denied; (4) `checkRateLimit` else error; (5) Zod parse params else error; (6) if tool is confirmation-required and `!confirmed` → return `confirm_required`; (7) `tool.execute` (try/catch → typed error, never throw); (8) audit `succeeded`/`failed`; (9) return outcome.
